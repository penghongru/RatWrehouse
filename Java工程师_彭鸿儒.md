     * 过程:  

       在排查中发现,并发数并没有达到超出服务器的承受上限,且问题发生在DB层,后排查业务代码未发现明显问题,只找到了几处疑似仍未能解决问题,后检查日志梳理调用顺序发现问题所在.

     * 解决:

       将耦合的服务改用消息中间件异步处理,并把G端和B端的用户系统分开部署到不同的服务器上,解决问题上线.后续对系统业务架构的持续优化不一一赘述

    * 事件2

      智能排课服务开发,随着公司发展用户数量增加,教研部门排课压力急剧增大,出现了几次课程错排的情况.应需求开发智能排课服务

      * 需求:

        根据目标班级进行排课推荐并自动生成推荐排课

        排课所选书籍需根据书籍的阅读分级对应年级的年龄阶段

        排课所选书籍应优先考虑本地库存

        排课所选书籍应是当前班级未读过的

        若本地库存不足以满足需求,则考虑异地库存,需考虑物流费用及时间,根据时间的选择物流和异地库存

      * 实现:
        1. 创建条件接口,声明条件需求:设置班级属性,根据班级属性获得限制条件

        2. 创建库存接口,对接库存系统,获取库存信息

        3. 创建匹配策略,并以责任链形式实现多个策略顺序执行

        4. 本地策略,在本地库存中查找匹配的书籍,若找到则返回结果,若无法找到则调用异地策略

        5. 异地策略,对接物流管理系统,根据物流管理系统获取的异地库存状态,根据库存剩余状态和时间要求进行排序后存储到数组,对数组遍历获取到结果后返回数据,在排课成功后发送通知到物流管理部和财务部.
